# Reproducible Execution Cases

Below are execution cases generated by the LeviathanMatrix AEP open-core
runtime. Each case is reproducible from the CLI and demonstrates policy
evaluation, execution pass issuance, capital capsule binding, receipt
generation, and post-execution review.

These cases are not claims of third-party production usage. They are
runtime-generated AEP execution records designed to show how the open-core
system behaves under different execution conditions.

Runtime identifiers such as `case_id`, `issuance_id`, `capsule_id`,
`execution_id`, and `tx_id` are generated per run, so exact IDs may differ when
the same command is executed again.

This case demonstrates the clean lifecycle: a natural-language request becomes
a governed execution path with authorization, issuance, capsule binding,
receipt generation, and review.

```bash

  --text "buy 1 USDC of SOL" \
```

- The lifecycle ends with post-execution review.

## Case 2: Fail-Closed Notional Protection

This case demonstrates that AEP is not a prompt wrapper. A request that exceeds
policy limits is blocked before execution.

Run:

```bash
python scripts/aep_cli.py run-text \
  --text "buy 1000000 USDC of SOL" \
  --agent-id demo-agent
```

Observed output:

```json
{
  "ok": false,
  "producer": {
    "company": "LeviathanMatrix",
    "product": "AEP",
    "project": "LeviathanMatrix AEP Open Core",
    "component": "cli_summary",
    "version": "0.1.0",
    "spec_id": "leviathanmatrix.aep.open-core.v1",
    "implementation": "leviathanmatrix-aep-open-core"
  },
  "spec_id": "leviathanmatrix.aep.open-core.v1",
  "case_id": "case_req_56cedc240f446a76",
  "authorization": {
    "status": "DENIED",
    "issuance_id": "iss_4c330dfd68f9eb7efc6c",
    "capsule_id": ""
  },
  "execution": {
    "status": "BLOCKED",
    "execution_id": "",
    "tx_id": ""
  },
  "receipt": {
    "status": ""
  },
  "review": {
    "status": "FAILED"
  }
}
```

What this demonstrates:

- AEP can reject execution even when the input is a valid natural-language
  trading request.
- No capital capsule is issued for the denied action.
- The execution adapter is not called after authorization fails.
- The final state remains reviewable instead of disappearing as a silent
  failure.

## Why This Matters

The point is not that an agent can produce a paper trade. The point is that an
agent cannot move through the execution path unless the request becomes:

- policy-bound;
- pass-bound;
- notional-bound;
- execution-mode-bound;
- receipt-producing;
- reviewable.

For Web4 agents, this is the difference between an agent that merely says it is
safe and an execution system that can show how authority was created, bounded,
used, blocked, or reviewed.
